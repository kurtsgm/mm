## 2. 框架層 vs 內容層（決定優先序的分界）

- **框架層**＝系統，做一次全體內容受益（戰鬥 UI、狀態異常 mechanic、引擎能力）。
- **內容層**＝用既有框架產出的資料/美術（map、monster、quest、vendor、portrait、室內地圖）。

你的目標是「先把**框架**做完，之後直接拓展內容」。盤點後，**真正還算框架的缺口只有戰鬥相關兩項**；城鎮室內、更多怪/任務都已落到內容層。

## 4. 框架「完工」的定義（Exit Criteria）

當以下全部成立，即可宣告**框架完成、轉內容拓展期**：

- [ ] 戰鬥有真正可玩介面（非 placeholder、版面比例式）。
- [ ] 戰鬥行動齊全：攻擊 / 防禦 / 法術 / **道具** / 逃跑。
- [ ] 狀態異常含至少一種 DoT 與一種控制類（暈眩/沉默），且能被法術與怪物技能引用。
- [ ] 「新增一個含室內店的城鎮 / 一隻新怪 / 一條新任務」全程**只需編資料**，不需改引擎。（最後一項其實**現在已大致成立**。）

## 5. 建議路線

**先做 #1 戰鬥 UI（含 #2 道具行動）→ 再做 #3 狀態異常 → 穿插 #4 接通 level01（XS）→ 之後轉內容期**（室內城鎮、更多怪/任務、怪物美術）。

理由：

1. #1 是核心迴圈裡唯一的 placeholder，且順手修掉「寫死像素」違反 UI 規範的債。純 presentation 層、戰鬥邏輯不動，**風險低**。
2. #1、#3 都動戰鬥，但 #1 是呈現、#3 是規則；**先呈現再規則**比較好驗證，#3 的效果也要靠 #1 的 UI 才看得出來。
3. #4 是 XS 隨時可插隊的快速勝利。
4. 做完 #1–#3，「拓展城鎮/怪物/任務」就純粹是編資料 + 美術，符合你「框架做完再拓展內容」的策略。
