# MM3 風格格子第一人稱 RPG — 技術架構設計

- **日期**：2026-06-24
- **狀態**：架構設計已核可，待產出實作計畫
- **範圍**：技術架構（不含具體遊戲內容與美術風格，那些後續另案處理）

## 目標

做一款類似 *Might and Magic III* 風格的遊戲：

- **格子式移動**：世界是離散的 N×N 格，移動以一格為單位
- **第一人稱視角**：玩家從隊伍視角看世界
- **四向面向**：只朝東南西北，轉向以 90° 為單位

本文件只規劃**技術架構**。遊戲內容（怪物、地圖、劇情、平衡）與美術風格屬於後續工作，架構刻意設計成讓這些內容能在不改引擎程式碼的前提下加入。

## 核心技術決策

| 層面 | 決定 | 理由 |
|------|------|------|
| 目標平台 | 桌面 PC（Windows / macOS / Linux） | 效能最佳、最接近傳統 RPG 體驗 |
| 引擎 | Godot 4 | 免費開源、輕量、學習曲線平緩、社群活躍 |
| 程式語言 | GDScript | 與編輯器/節點整合最佳、迭代最快、文件資源最多 |
| 渲染 | 混合：3D 模組化場景 + 2D billboard 怪物/NPC | 最適合用 AI 生素材，且最忠於 MM3 原版（原版怪物即 2D 立繪） |
| 架構取向 | 資料驅動（引擎/內容分離） | 內容後續再做，加內容不應碰程式碼 |

### 渲染決策補充：為何混合而非真 3D

關鍵在 AI 生成素材在 2D 與 3D 間的成熟度落差（截至 2026 初）：

- **AI 生 2D 怪物圖**：成熟、可控、便宜、風格一致性可調，且**不需綁骨架、不需 3D 動畫**，生出即用。
- **AI 生 3D 怪物模型**：可生靜態 mesh，但拓樸雜亂、難綁骨架；**動畫（idle/攻擊/死亡）幾乎要手工**，對 solo dev 是大坑。

因此採「場景 3D、怪物 2D」：場景吃現代 3D 光影與空間感、且可用模組化磚塊（靜態、好做、可買可生）；怪物吃 2D 的 AI 生成便利與復古味。額外好處：MM3 原版怪物本就是 2D 立繪，此作法最忠實。代價是 2D 怪物從側面是「紙片」，但 billboard 永遠面向相機，格子制第一人稱下玩家看不到側面，影響可忽略。

## 玩法預設（MM3 忠實預設，後續可調）

以下偏向玩法，先採 MM3 忠實預設，架構保留調整彈性：

- **隊伍制**：一隊多名角色（非單一主角），資料模型以「隊伍 + 多角色」設計
- **回合制戰鬥**：遭遇怪物時切換到回合制，保留之後加「即時/回合切換」的彈性
- **離散格子世界**：世界為 N×N 格，面向僅四向，移動/轉向以補間動畫平滑過渡

## 三層分離架構

```
┌─────────────────────────────────────────────┐
│  Content 內容層 — 純資料，零邏輯               │
│  地圖 / 怪物 / 道具 / 法術 / 職業              │
│  → .tres Resource 檔 + AI 生 2D 貼圖 + 3D 磚塊  │
└─────────────────────────────────────────────┘
              ↑ 執行時載入
┌─────────────────────────────────────────────┐
│  Engine 引擎層 — 通用系統，完全不含具體內容     │
│  Grid / Movement / Combat / Party / Spell /   │
│  Inventory / Encounter / Save                 │
└─────────────────────────────────────────────┘
              ↑ 驅動
┌─────────────────────────────────────────────┐
│  Presentation 呈現層 — Godot 節點              │
│  3D 場景渲染 / 2D billboard / UI / 輸入        │
└─────────────────────────────────────────────┘
```

**核心不變式**：之後加一隻怪、加一張地圖 = 加一個資料檔，完全不碰引擎層程式碼。

### 各層職責與依賴

- **Content 內容層**：只有資料（Godot `Resource` / `.tres`）與素材檔。不依賴任何其他層。可被 Engine 層讀取。
- **Engine 引擎層**：純 GDScript 邏輯，盡量不直接依賴 Godot 視覺節點（便於單元測試）。依賴 Content 的資料結構（透過自訂 Resource 類別介面）。
- **Presentation 呈現層**：Godot 節點與場景，訂閱/驅動 Engine 層。依賴 Engine 與 Content；Engine 不反向依賴 Presentation。

## 核心模組

| 模組 | 職責 | 型態 |
|------|------|------|
| `GameState` | 全域狀態：當前隊伍、地圖、旗標/進度 | autoload 單例 |
| `MapManager` | 載入/切換地圖，提供格子查詢（能否通行、有無牆/事件） | autoload |
| `PlayerController` | 格子移動與轉向：輸入 → 撞牆檢查 → 補間動畫 → 更新座標 | 系統 |
| `EncounterSystem` | 踩怪格 / 隨機遭遇 → 觸發戰鬥 | 系統 |
| `CombatSystem` | 回合制流程：行動順序、攻擊、施法、逃跑 | 系統 |
| `Party` / `Character` | 隊伍與角色資料模型（屬性、HP/SP、裝備、狀態） | 資料類別 |
| `Inventory` / `Equipment` | 道具與裝備管理 | 系統 |
| `SpellSystem` | 法術定義與施放效果 | 系統 |
| `SaveSystem` | 序列化 GameState 存讀檔 | autoload |

## 資料驅動內容（Godot 自訂 Resource）

用 Godot 的 `Resource`（`.tres` 檔）定義所有內容。自訂類別（定義放 `res://resources/`）：

| Resource 類別 | 主要欄位 |
|---------------|----------|
| `MonsterDef` | 名稱、2D 貼圖、屬性、攻擊、掉落表、AI 類型 |
| `ItemDef` | 名稱、圖示、類型、加成、價值 |
| `SpellDef` | 名稱、流派、消耗、效果參數 |
| `MapData` | 格子尺寸、tile 陣列（地板/牆/特殊）、怪物配置、事件、遭遇表 |
| `CharacterClass` | 職業定義（成長、可用法術流派等） |
| `TileSet3D` | tile 類型 → 3D 磚塊 mesh 的對應 |

地圖編輯起步策略：先以 in-editor 可編輯的 `.tres`（或簡單 script 產生）建立 `MapData`，未來視需求再做專用地圖編輯工具或匯入外部格式。

## 格子與移動系統

- **世界表示**：2D 陣列，每格存「地板類型、四向牆旗標 (N/E/S/W)、佔用者/事件 id」
- **玩家狀態**：`grid_pos: Vector2i` + `facing: enum {N,E,S,W}`
- **移動流程**：輸入（前進/後退/平移/轉向）→ 檢查目標格牆壁碰撞 → 補間相機位置/旋轉（約 0.15–0.25 秒）→ 更新 `grid_pos`；補間期間鎖輸入（佇列或忽略）
- **場景幾何**：用 Godot 內建 **GridMap** 節點（專為格子場景設計）從 MeshLibrary 組牆與地板；由 `MapData` 驅動填入

## 混合渲染

- **場景**：3D 模組化磚塊（GridMap / 實例化 tile 場景）+ Godot 真實 3D 光影（即時或烘焙）
- **怪物/NPC**：`Sprite3D` 開 billboard，貼圖來自 `MonsterDef`（AI 生）。戰鬥時固定排在相機前方
- **UI**：Godot Control 節點（CanvasLayer）

## Godot 場景結構

```
Main (root)
├── World3D (Node3D)
│   ├── GridMap            ← 場景幾何（由 MapData 填入）
│   ├── PartyCamera        ← 第一人稱相機，鎖格子
│   ├── Entities           ← 怪物/NPC billboards
│   └── Lights
├── HUD (CanvasLayer)      ← 隊伍狀態、指北針、訊息列
├── CombatLayer            ← 戰鬥 UI
└── Menus (CanvasLayer)    ← 背包、角色、存讀檔
```

## 專案目錄結構

```
res://
├── engine/        # 通用系統 GDScript（無內容）：grid/ combat/ party/ save/ ...
├── content/       # 資料層 .tres + 素材：monsters/ items/ spells/ maps/ classes/
├── presentation/  # 場景、UI、shader、3D 磚塊 MeshLibrary：world/ ui/ tiles/
├── resources/     # 自訂 Resource class 定義（MonsterDef.gd 等）
└── tests/         # GUT 單元測試
```

## 存檔系統

序列化 GameState（隊伍、座標、地圖旗標、背包）→ 存到 `user://` 目錄，支援多存檔槽。序列化格式以 JSON 為主（可讀、好除錯）；地圖等大型靜態內容不入存檔，存檔只記「目前進度狀態」並以 id 參照內容資料。

## 測試策略

- 引擎層為純邏輯 → 用 **GUT（Godot Unit Test）** 對以下做單元測試並採 TDD：
  - 格子數學與撞牆判定
  - 戰鬥結算（傷害公式、行動順序）
  - 存讀檔 roundtrip（序列化 → 反序列化一致）
- 呈現層（渲染、UI、輸入）靠手動/整合測試。
- 引擎層刻意與 Godot 視覺節點解耦，以利測試獨立執行。

|--------|------|

## 非目標（本階段不做）

- 具體遊戲內容（怪物清單、地圖設計、劇情、數值平衡）
- 美術風格定調與素材正式產製
- 多人連線
- 行動裝置 / 主機平台移植
- 專用視覺化地圖編輯器（起步用資料檔，視需求再做）
